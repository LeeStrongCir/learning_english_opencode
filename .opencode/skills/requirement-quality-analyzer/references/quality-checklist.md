# 需求质量检查清单

本文件是 `requirement-quality-analyzer` 的详细检查清单，在执行质量检查时参考。

## 1. 完整性检查清单

### 1.1 功能描述
- [ ] 每个功能是否有明确的输入和输出？
- [ ] 是否说明了触发条件（用户操作、系统事件、定时任务）？
- [ ] 是否说明了前置条件（用户需要登录、需要有权限等）？
- [ ] 是否说明了后置条件（数据变化、状态变化、通知发送）？

### 1.2 异常处理
- [ ] 是否说明了各种异常情况的处理方式？
  - 网络异常（断网、超时、弱网）
  - 服务异常（第三方服务不可用、数据库异常）
  - 数据异常（空数据、脏数据、格式错误、超长数据）
  - 权限异常（未登录、无权限、权限过期）
- [ ] 是否说明了错误提示文案？
- [ ] 是否说明了异常后的恢复机制？

### 1.3 边界条件
- [ ] 是否定义了数值范围（最小值、最大值）？
- [ ] 是否定义了字符串长度限制？
- [ ] 是否定义了数量限制（最多/最少）？
- [ ] 是否定义了时间范围（有效期、超时时间）？
- [ ] 是否考虑了空值、零值、负值的情况？

### 1.4 数据约束
- [ ] 是否定义了数据格式（日期格式、手机号格式、邮箱格式）？
- [ ] 是否定义了唯一性约束（用户名唯一、订单号唯一）？
- [ ] 是否定义了必填/选填字段？
- [ ] 是否定义了数据关联关系（外键、级联删除）？

### 1.5 权限控制
- [ ] 是否说明了谁能操作（角色、权限）？
- [ ] 是否说明了谁能查看（数据可见范围）？
- [ ] 是否说明了越权处理方式？

### 1.6 状态流转
- [ ] 是否定义了所有可能的状态？
- [ ] 是否定义了状态之间的转换规则？
- [ ] 是否定义了非法状态转换的处理？
- [ ] 是否说明了状态变更的触发条件？

## 2. 清晰度检查清单

### 2.1 模糊词汇识别
检查以下词汇是否出现，如果出现，要求量化：

| 类别 | 模糊词汇 | 量化建议 |
|------|----------|----------|
| 时间 | 快速、及时、尽快、实时、秒级 | P95 < X ms、X 秒内完成 |
| 程度 | 大概、可能、尽量、基本、较好 | 明确具体标准 |
| 数量 | 一些、多个、少量、大量 | 具体数字或范围 |
| 体验 | 友好、美观、流畅、简洁 | 可衡量的指标 |
| 其他 | 等、相关、适当、合理 | 明确列举或定义 |

### 2.2 歧义识别
- 同一表述是否有多种理解？
- 专业术语是否有定义？
- "或"是包含还是互斥？
- "以上/以下"是否包含本数？

## 3. 一致性检查清单

### 3.1 内部一致性
- [ ] 同一需求中是否有矛盾描述？
- [ ] 同一概念是否使用了不同名称？
- [ ] 数据格式、单位是否统一？
- [ ] 优先级描述是否一致（前面说重要，后面说可选）？

### 3.2 外部一致性
- [ ] 与其他需求是否冲突？
- [ ] 与现有系统行为是否一致？
- [ ] 与产品整体定位是否一致？
- [ ] 与技术架构是否匹配？

## 4. 可测试性检查清单

### 4.1 验收标准
- [ ] 每条需求是否有明确的验收标准？
- [ ] 验收标准是否可量化？
- [ ] 验收标准是否可验证（能设计出具体的测试用例）？

### 4.2 测试环境
- [ ] 是否需要特殊的测试环境？
- [ ] 是否需要第三方服务的测试账号或 Mock？
- [ ] 是否需要特定的测试数据？

### 4.3 主观判断
- [ ] 是否存在无法自动化验证的主观判断？
- [ ] 是否需要人工评审或用户验收？

## 5. 风险检查清单

### 5.1 技术风险
- [ ] 是否引入了新技术或新框架？
- [ ] 是否有性能瓶颈预期？
- [ ] 是否有并发处理需求？
- [ ] 是否有数据迁移需求？

### 5.2 依赖风险
- [ ] 是否依赖外部系统或第三方服务？
- [ ] 依赖的服务是否稳定？
- [ ] 是否有降级方案？
- [ ] 是否有 Mock 或替代方案？

### 5.3 合规风险
- [ ] 是否涉及用户隐私数据？
- [ ] 是否符合行业规范（金融、医疗、教育等）？
- [ ] 是否符合数据保护法规（个人信息保护法、GDPR 等）？
- [ ] 是否需要安全审计？

### 5.4 变更风险
- [ ] 需求是否稳定？是否可能频繁变更？
- [ ] 开发时间是否紧张？
- [ ] 测试时间是否充足？
- [ ] 是否影响现有功能？

## 评级参考

### A - 优秀
- 所有检查项均通过
- 无模糊表述
- 无矛盾点
- 所有需求均可验证
- 可直接进入开发

### B - 良好
- 1-2 项非关键检查项未通过
- 1-3 处模糊表述
- 1-2 处轻微不一致
- 1-2 条需求验收标准不明确
- 有小问题，可边做边改

### C - 合格
- 3-5 项关键检查项未通过
- 4-7 处模糊表述
- 3-5 处不一致
- 3-5 条需求无法设计用例
- 建议补充关键信息后再开发

### D - 不合格
- 5 项以上检查项未通过
- 8 处以上模糊表述或核心逻辑模糊
- 6 处以上不一致或核心逻辑矛盾
- 大部分需求无法验证
- 必须返工
