# 覆盖率评定最佳实践

## 1. 需求覆盖率评定

### 需求追踪矩阵建立方法

1. **从需求文档提取需求清单**
   - 每个需求分配唯一 ID（如 REQ-001）
   - 记录需求描述、优先级、所属模块

2. **建立需求→用例映射**
   - 在每个用例中标注关联的需求 ID
   - 使用双向追踪：需求→用例 和 用例→需求

3. **检查覆盖完整性**
   - 每个需求至少 1 个正向用例
   - 每个需求至少 1 个反向/异常用例
   - 高优先级需求应有更多用例覆盖

### 常见漏测场景

- 需求变更未同步更新用例
- 隐性需求（如性能、安全）被忽略
- 第三方集成需求未单独测试

## 2. 功能维度覆盖

### 6 维度检查清单

| 维度 | 检查点 | 示例 |
|------|--------|------|
| 正向流程 | 正常输入是否测试 | 正确用户名密码登录 |
| 反向流程 | 异常输入是否测试 | 错误密码、空用户名 |
| 边界值 | 临界点是否测试 | 密码长度 6 位、20 位 |
| 权限/角色 | 不同角色是否测试 | 管理员、普通用户、游客 |
| 数据状态 | 各种数据状态是否测试 | 空列表、大数据量、脏数据 |
| 兼容性 | 多端是否测试 | Chrome、Firefox、移动端 |

### 评分标准

- **6/6 维度**：100% - 优秀
- **5/6 维度**：83% - 良好
- **4/6 维度**：67% - 合格
- **<4/6 维度**：需补充

## 3. 场景路径覆盖

### 场景识别方法

1. **用户故事地图**：从用户视角梳理操作流程
2. **流程图分析**：识别分支点和决策点
3. **状态机分析**：识别所有可能的状态转换

### 路径分类

| 类型 | 优先级 | 覆盖要求 |
|------|--------|----------|
| 主流程 | P0 | 必须 100% 覆盖 |
| 分支流程 | P1 | 应该覆盖 |
| 异常流程 | P1 | 应该覆盖 |
| 边缘场景 | P2/P3 | 评估后决定 |

## 4. 代码覆盖率

### 工具推荐

| 语言 | 工具 |
|------|------|
| Python | Coverage.py, pytest-cov |
| JavaScript | Istanbul/nyc, Jest |
| Java | JaCoCo |
| Go | go test -cover |

### 解读指南

- **行覆盖率**：多少行代码被执行
- **分支覆盖率**：多少条件分支被执行
- **函数覆盖率**：多少函数被调用

### 注意事项

- 高覆盖率 ≠ 高质量用例
- 用例可能执行了代码但没有正确断言
- 结合需求覆盖率一起评估

## 5. 缺陷回归覆盖

### 回归用例管理

1. **每个修复的 bug 创建回归用例**
2. **用例中标注关联的 bug ID**
3. **定期执行回归测试集**

### 回归测试策略

- **P0 bug**：必须回归，加入自动化
- **P1 bug**：应该回归
- **P2/P3 bug**：选择性回归

---

## 6. 用例质量审查

### 审查清单

| 检查项 | 标准 |
|--------|------|
| 前置条件 | 明确、可验证 |
| 测试步骤 | 编号清晰、可执行 |
| 预期结果 | 具体、可断言 |
| 独立性 | 不依赖其他用例 |
| 可维护性 | 需求变更时易更新 |

### 模糊描述示例（应避免）

- ❌ "系统应该正常工作"
- ❌ "页面显示正常"
- ❌ "数据正确保存"

### 具体描述示例（推荐）

- ✅ "返回 HTTP 200，响应体包含 user_id 字段"
- ✅ "页面显示 3 个商品卡片，每个包含名称和价格"
- ✅ "数据库 users 表新增一条记录，email 字段值为 'test@example.com'"
