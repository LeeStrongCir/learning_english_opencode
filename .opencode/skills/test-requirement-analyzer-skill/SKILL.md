# 需求测试分析专家

基于 10+ 年测试经验沉淀的 AI 分析技能。

## 核心能力

1. **质疑思维**：发现需求本身的问题和漏洞
2. **场景化分析**：从用户视角还原完整使用场景
3. **风险嗅觉**：识别高风险区域，指导资源分配

## 分析流程

### 第一步：理解需求

阅读需求文档或需求描述，提取：
- 业务背景和目标用户
- 核心功能点
- 输入输出
- 业务规则
- 依赖关系（第三方服务、外部系统）

### 第二步：测试目标识别

明确要验证什么，每个目标包含：
- 目标描述（用 [OBJ-XXX] 编号）
- 优先级（P0/P1/P2）
- 关联的需求点
- 验收标准

**优先级评估依据**：业务影响 × 使用频率 × 实现复杂度

| 级别 | 含义 | 示例 |
|------|------|------|
| P0 | 核心功能，出问题影响大，必须测试 | 登录、支付、数据保存 |
| P1 | 重要功能，应该测试 | 搜索、筛选、导出 |
| P2 | 边缘功能，可以抽样测试 | UI 美化、提示文案 |

### 第三步：测试范围界定

明确"测什么"和"不测什么"：

**范围内**：本次迭代新增或修改的功能点
**范围外**：明确排除的内容（同样重要！避免测试蔓延）

### 第四步：风险识别

从四个维度识别风险：

| 维度 | 关注点 |
|------|--------|
| 技术风险 | 实现复杂度、新技术引入、性能瓶颈 |
| 依赖风险 | 第三方服务不稳定、外部接口变更 |
| 变更风险 | 需求不稳定、时间紧迫、范围蔓延 |
| 数据风险 | 数据安全、隐私合规、数据迁移 |

每个风险要说明：描述、影响程度（高/中/低）、缓解建议。

### 第五步：需求问题清单

用"质疑思维"审视需求，找出：

| 问题类型 | 特征 |
|----------|------|
| 模糊描述 | "大概"、"可能"、"尽量"、"快速响应"等不量化表述 |
| 缺失信息 | 缺少异常处理说明、边界条件、错误提示 |
| 矛盾点 | 不同地方描述不一致 |
| 需要澄清 | 业务逻辑不明确、分支场景未覆盖 |

### 第六步：测试建议

推荐：
- 测试类型（功能/接口/性能/安全/兼容性）
- 关键测试场景（正常流程、异常流程、边界场景、并发场景）
- 测试数据准备建议

## 输出格式

分析报告使用以下结构：

```markdown
# 需求测试分析报告

## 1. 需求概述
- **项目名称**: {名称}
- **分析时间**: {时间}

## 2. 测试目标
- [OBJ-001] {目标描述} **（优先级：P0）**
  - 验收标准：{标准 1}
  - 验收标准：{标准 2}

## 3. 测试范围
### 3.1 范围内
- {功能点 A}
- {功能点 B}

### 3.2 范围外
- {明确不测试的内容}

## 4. 优先级评估
### P0（必须测试）
- {核心功能}

### P1（应该测试）
- {重要功能}

### P2（可以测试）
- {边缘功能}

## 5. 风险识别
- 🔴 **{风险类别}**: {风险描述}
  - 影响：{影响说明}
  - 缓解：{缓解建议}

## 6. 需求问题清单
- [Q-001] ❓ {问题类型}: {问题描述}
  - 位置：{需求中的位置}

## 7. 测试建议
**推荐测试类型**:
- {类型 1}
- {类型 2}

**关键场景**:
- ✅ {正常流程场景}
- ⚠️ {异常流程场景}
- 📏 {边界场景}

## 8. 分析总结
{200 字以内的总结}

---
> 💡 **提示**: 本分析由 AI 生成，请务必人工 review 后使用。
> 重点关注"需求问题清单"，这些问题需要在需求评审会上澄清。
```

## 示例

**输入**：
> 用户登录功能：支持账号密码登录和手机验证码登录。账号密码登录时，密码错误 3 次锁定 30 分钟。手机验证码有效期 5 分钟。

**输出要点**：
- 测试目标：验证两种登录方式的正确性（P0）、验证锁定机制（P0）、验证验证码有效期（P1）
- 风险：依赖短信服务（高风险）、锁定机制可能被绕过（中风险）
- 问题：没有说明锁定后的提示文案、没有说明验证码发送频率限制

## 注意事项

1. **人机协作**：AI 分析 + 人工 review 是最佳实践
2. **尽早使用**：需求评审前先用此 skill 分析，带着问题参会
3. **持续优化**：根据实际 bug 反馈调整分析重点
4. **灵活适配**：根据需求规模调整分析深度，小需求可以简化输出
