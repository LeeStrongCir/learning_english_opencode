## 缺陷详情

### DEF-001: 自动化脚本元素定位器与页面结构不匹配

**基本信息**:

| 属性 | 值 |
|------|-----|
| 缺陷 ID | DEF-001 |
| 严重程度 | 🔴 高 (High) |
| 功能模块 | 自动化框架 / Page Object |
| 发现日期 | 2026-04-18 |
| 状态 | Open |
| 影响用例数 | 14 个 |
| 错误类型 | Element locator mismatch |

**问题描述**:

自动化脚本 `pages/login_page.py` 中的元素定位器与阿里云登录页面当前实际 DOM 结构不匹配，导致 14 个测试用例执行时抛出 `Element locator mismatch` 错误。

**具体不匹配项**:

| 定位器名称 | 脚本中的定位器 | 问题描述 |
|-----------|--------------|---------|
| 账号输入框 | `#fm-login-id` | 页面已不再使用此 ID，实际使用 `input[placeholder='请输入']` |
| 密码输入框 | `#fm-login-password` | 页面已不再使用此 ID |
| 登录按钮 | `#fm-login-submit` | 页面已不再使用此 ID，实际使用 `button:has-text('立即登录')` |
| 手机号输入（短信登录） | `#mobile` | 页面元素动态生成，ID 不固定 |
| 验证码输入（短信登录） | `#mobilecode` | 页面元素动态生成，ID 不固定 |
| 错误提示 | `.error-msg` | 页面使用动态 class 名，固定 class 不匹配 |

**复现步骤**:

1. 运行自动化测试脚本：`pytest tests/test_login.py`
2. 观察 TC-LOGIN-001、TC-LOGIN-002 等用例执行
3. 用例在等待元素可见时超时，抛出 `Element locator mismatch` 错误

**预期**: 定位器能正确匹配页面元素
**实际**: 定位器无法找到元素，测试失败

**影响**:
- 14 个测试用例（41%）直接失败
- 无法验证登录功能、安全测试等核心功能
- 自动化测试脚本失去可用性

**建议修复方案**:
1. 使用更稳定的选择器策略：
   - 优先使用 `data-testid` 属性（需前端配合添加）
   - 使用 `placeholder` 属性定位输入框
   - 使用 `text` 内容定位按钮和标签
2. 更新 `LOCATORS` 字典中的所有定位器
3. 增加定位器健康检查机制，在测试执行前验证所有定位器有效性

**修复优先级**: P0（阻塞性缺陷）

### DEF-002: 生产环境验证码拦截自动化登录流程

**基本信息**:

| 属性 | 值 |
|------|-----|
| 缺陷 ID | DEF-002 |
| 严重程度 | 🔴 高 (High) |
| 功能模块 | 安全机制 / 验证码 |
| 发现日期 | 2026-04-18 |
| 状态 | Open |
| 影响用例数 | 约 8 个 |
| 错误类型 | Captcha required |

**问题描述**:

阿里云登录页面检测到自动化测试工具（Playwright）特征后，触发验证码验证机制（滑块验证/图形验证），导致自动化登录流程无法完成。

**复现步骤**:

1. 使用 Playwright 打开 `https://account.aliyun.com/login/login.htm`
2. 输入账号和密码
3. 点击"立即登录"按钮
4. 页面弹出验证码验证（滑块或图形）
5. 自动化脚本无法处理验证码，登录流程中断

**预期**: 自动化测试能正常完成登录流程
**实际**: 验证码拦截导致登录失败

**影响**:
- 约 8 个测试用例（24%）因验证码拦截而失败
- 所有需要完成登录流程的用例均受影响
- 核心功能（登录成功跳转、登录失败提示等）无法验证

**建议修复方案**:
1. **短期**：申请阿里云测试环境，配置验证码白名单或关闭验证码
2. **中期**：实现验证码自动识别（如使用 OCR 或第三方验证码服务）
3. **长期**：与阿里云安全团队协作，建立自动化测试专用通道

**修复优先级**: P0（阻塞性缺陷）

### DEF-003: 测试数据使用模拟账号无法完成登录验证

**基本信息**:

| 属性 | 值 |
|------|-----|
| 缺陷 ID | DEF-003 |
| 严重程度 | 🟡 中 (Medium) |
| 功能模块 | 测试数据管理 |
| 发现日期 | 2026-04-18 |
| 状态 | Open |
| 影响用例数 | 约 6 个 |
| 错误类型 | Invalid test credentials |

**问题描述**:

测试数据文件 `fixtures/test-data.json` 中配置的账号均为模拟账号（如 `13800138000`、`test@example.com`、`test_user_001`），这些账号在阿里云生产环境中不存在，导致需要真实凭证的登录用例无法完成验证。

**复现步骤**:

1. 查看 `fixtures/test-data.json` 中的 `valid_accounts` 配置
2. 使用其中的账号尝试登录阿里云生产环境
3. 登录失败，提示账号不存在或密码错误

**预期**: 测试数据中包含有效的测试账号
**实际**: 所有测试账号均为模拟数据，无法用于生产环境

**影响**:
- 约 6 个测试用例（18%）因账号无效而失败
- 正常登录流程无法验证
- 登录成功后的跳转行为无法验证

**建议修复方案**:
1. 申请阿里云测试环境，配置专用测试账号
2. 建立测试数据与环境对应的配置机制（dev/test/prod）
3. 使用环境变量或配置文件管理测试数据，避免硬编码

**修复优先级**: P1

### DEF-004: 缺少测试环境配合无法完成完整测试

**基本信息**:

| 属性 | 值 |
|------|-----|
| 缺陷 ID | DEF-004 |
| 严重程度 | 🔴 高 (High) |
| 功能模块 | 测试环境 |
| 发现日期 | 2026-04-18 |
| 状态 | Open |
| 影响用例数 | 全部 34 个 |
| 错误类型 | Missing test environment |

**问题描述**:

本次测试直接在阿里云生产环境执行，缺少专用的测试环境配合。生产环境的限制（验证码、安全策略、真实账号要求）导致大部分测试用例无法正常执行。

**影响范围**:

| 影响项 | 说明 |
|--------|------|
| 验证码机制 | 生产环境触发验证码，自动化无法处理 |
| 账号锁定 | 多次失败登录可能触发真实账号锁定 |
| 安全策略 | 生产环境安全策略更严格，影响测试执行 |
| 数据隔离 | 测试数据与生产数据混合，存在风险 |

**预期**: 有独立的测试环境，支持自动化测试执行
**实际**: 直接使用生产环境，受限于安全策略

**影响**:
- 全部 34 个测试用例受影响
- 测试通过率为 0%
- 无法完成任何有效功能验证

**建议修复方案**:
1. 与阿里云团队协调，获取测试环境访问权限
2. 测试环境应具备：
   - 验证码白名单或关闭验证码
   - 预配置的测试账号
   - 与生产环境一致的功能
3. 建立测试环境管理规范

**修复优先级**: P0（阻塞性缺陷）

### DEF-005: 登录方式切换标签文本与脚本预期不一致

**基本信息**:

| 属性 | 值 |
|------|-----|
| 缺陷 ID | DEF-005 |
| 严重程度 | 🟡 中 (Medium) |
| 功能模块 | UI 自动化 / 登录方式切换 |
| 发现日期 | 2026-04-18 |
| 状态 | Open |
| 影响用例数 | 3 个 |
| 错误类型 | Text mismatch |

**问题描述**:

自动化脚本中登录方式切换的标签文本与实际页面不一致：

| 切换项 | 脚本预期文本 | 实际页面文本 |
|--------|------------|-------------|
| 短信登录 | `text=短信登录` | `text=手机号登录` |
| 扫码登录 | `text=扫码登录` | 无独立标签（左侧二维码区域） |
| 密码登录 | `text=账密登录` | `text=账密登录`（正确） |

**复现步骤**:

1. 运行 `test_switch_to_sms_login` 用例
2. 脚本尝试点击 `text=短信登录`
3. 页面实际显示为 `text=手机号登录`
4. 元素查找超时，测试失败

**预期**: 标签文本与脚本预期一致
**实际**: 标签文本已变更，脚本未同步更新

**影响**:
- 3 个登录方式切换用例失败
- 无法验证登录方式切换功能
- 影响用户体验相关的测试覆盖

**建议修复方案**:
1. 更新 `login_page.py` 中的 `sms_login_tab` 定位器为 `text=手机号登录`
2. 重新评估扫码登录的切换方式（可能需要不同的交互方式）
3. 建立 UI 文本变更检测机制

**修复优先级**: P1
